# Human Texture Evaluation Rubric

调研日期：2026-06-03

Human Texture 的目标不是让文本骗过 AI detector，而是判断正文是否像“人在经历故事”。本 rubric 建议用于 Reviewer、人审对照、以及未来 Human Texture Engine 的 gate。

## 评分方式

每项 1-5 分：

- 1 分：明显系统展示或模板化，几乎没有人味。
- 2 分：有局部人味，但主要仍服务于机制/信息。
- 3 分：基本合格，有可见人物动机和场景承载。
- 4 分：人物、关系、场景、信息自然融合。
- 5 分：人味成为爽点的一部分，读者能记住人的状态和余波。

建议门禁：

- 单章平均低于 3.5：不得进入 Polisher。
- 任一核心项低于 3：回到对应层修正。
- “是否仍像系统展示”低于 3：优先回 Planner/Writer，不做语言润色硬救。

## 1. 人物可信度

定义：

- 人物是否像有过去、身份、利害关系和生活惯性的人，而不是剧情按钮。

好文本表现：

- 人物行动符合其处境和认知限制。
- 配角有自己的小目标，不只围绕主角功能移动。
- 人物会误判、怕丢脸、顾及旁人眼光。

坏文本表现：

- 配角只负责提示、反对、作证、解释。
- 人物一开口就说剧情需要的话。
- 角色没有名字、习惯、关系位置或生活痕迹。

可审查问题：

- 这场戏里每个重要人物想要什么？
- 有没有人想要的东西和主线目标不完全一致？
- 如果删掉某人，是否只少一个功能？

可落地 Skill：

- Planner：为每个关键角色生成 `private_want` 与 `social_position`。
- Writer：写行动时体现认知限制和生活惯性。
- Reviewer：检查“功能件角色”并要求补 private agenda。

## 2. 人物私心 / 羞耻 / 犹豫

定义：

- 人物是否有不便明说的欲望、羞耻点、犹豫和自我保护。

好文本表现：

- 人物不总是把动机讲明白。
- 关键选择前有迟疑、绕开、嘴硬、补救或遮掩。
- 私心会影响判断，甚至造成小错误。

坏文本表现：

- 人物动机完全透明，像任务说明。
- 主角永远理性、正确、推进剧情。
- 羞耻和犹豫只被命名，没有改变行为。

可审查问题：

- 主角这一章最不想让别人看见什么？
- 哪个选择不是最优解，但符合人的自保？
- 有无因为羞耻/私心导致的微小损失？

可落地 Skill：

- Planner：新增 `shame_or_avoidance`。
- Writer：用沉默、错话、错动作表现。
- Reviewer：低分时退回 Planner，不建议 Polisher 硬改。

## 3. 情绪真实度

定义：

- 情绪是否通过身体、行为、认知偏差和后续关系体现，而不是只用情绪名词。

好文本表现：

- 情绪导致人物说错话、记错细节、回避目光、做出不完美选择。
- 情绪在下一场仍有残留。
- 情绪和人物身份有关，不是通用“恐惧/愤怒/痛苦”。

坏文本表现：

- 文本直接说“他害怕/愤怒/痛苦”，随后照常推进。
- 情绪被系统化为积分、杂质、蓝光后就结束。
- 群体情绪像数值条。

可审查问题：

- 这个情绪让人物失去了什么？
- 有没有身体或关系上的后续痕迹？
- 情绪是否具体到这个人物，而不是通用标签？

可落地 Skill：

- Planner：情绪必须绑定 `behavioral_consequence`。
- Writer：少用情绪名词，多写行为后果。
- Reviewer：检查“情绪命名但无后果”。

## 4. 关系摩擦

定义：

- 人物之间是否存在误会、亏欠、试探、信任变化和难以立刻修复的关系成本。

好文本表现：

- 人物关系会因为选择被消耗或拉近。
- 对话有潜台词，不只交换信息。
- 关系变化在后续章节被记住。

坏文本表现：

- 冲突后关系马上重置。
- 人物只在剧情需要时出现，没有日常黏性。
- 背叛、隐瞒、帮助没有后续账。

可审查问题：

- 这一章新增了哪笔关系债？
- 谁误会了谁？谁没有解释？
- 下一章关系会因此变难还是变容易？

可落地 Skill：

- Planner：维护 `relationship_debt_ledger`。
- Writer：写对话中的回避、试探和不完整解释。
- Reviewer：检查关系是否有“后续账”。

## 5. 场景生活质感

定义：

- 场景是否像本来就有人生活、工作、吃饭、受冻、排队、怕事，而不只是剧情发生平台。

好文本表现：

- 场景有原本的生活功能。
- 细节与人物处境、阶层、制度有关。
- 场景阻力会改变行动路径。

坏文本表现：

- 饭堂只用来传消息，矿洞只用来揭示设定。
- 地点名称可互换。
- 细节是装饰，不影响人物。

可审查问题：

- 这个地方没有主线时会怎样运转？
- 这里的普通人怕什么、图什么？
- 场景细节是否改变了人物行动？

可落地 Skill：

- Planner：生成 `scene_life_function` 与 `scene_resistance`。
- Writer：把生活阻力写进动作。
- Reviewer：检查场景是否只是信息触发器。

## 6. 非工具性细节

定义：

- 细节不直接服务解谜或设定，但能让人物、阶层、关系和生活状态变得可信。

好文本表现：

- 一个小物件、习惯、身体反应、称呼变化，让人感到人物存在。
- 细节不过度解释，不抢主线。
- 细节能在后续自然回响。

坏文本表现：

- 所有细节都指向伏笔、线索、设定。
- 为了“文学感”硬塞风景描写。
- 细节只出现一次，没有承载人物。

可审查问题：

- 这一章有没有一个无须解释但能记住人的细节？
- 它是否贴着人物处境，而不是装饰？
- 它是否会影响读者对关系或身份的理解？

可落地 Skill：

- Planner：每章最多指定 1-2 个 `non_instrumental_detail`。
- Writer：自然嵌入，不做说明。
- Reviewer：检查是否变成装饰性散文。

## 7. 信息露出自然度

定义：

- 设定、规则、真相是否通过行动、误读、交易、偷听、制度、物件、关系压力自然暴露。

好文本表现：

- 信息有载体和代价。
- 不同人物知道的信息不一致。
- 读者通过场景推断，而不是被公告喂给。

坏文本表现：

- 系统、古镜、权威人物长段说明。
- 主角“看懂”复杂机制但没有知识来源或误判。
- 信息没有误读风险。

可审查问题：

- 这条信息是谁说的？为什么他现在说？
- 是否可以由行动或冲突暴露？
- 信息露出有没有让人物付出代价？

可落地 Skill：

- Planner：为每条关键信息指定 `information_carrier` 与 `exposure_cost`。
- Writer：避免整段公告式解释。
- Reviewer：检查信息是否自然露出。

## 8. 代价是否造成真实后果

定义：

- 破局、吸收、隐瞒、撒谎、燃烧情绪等代价是否在身体、关系、资源、声誉、时间或心理上持续生效。

好文本表现：

- 代价改变下一场行动。
- 关系/身体/资源账被记住。
- 读者能感到“赢了但少了什么”。

坏文本表现：

- 代价只在当段疼一下。
- 章尾总结“他付出了代价”，但后续无影响。
- 破局仍然像免费升级。

可审查问题：

- 主角具体失去了什么？
- 下一章哪个选择因此变难？
- 旁人是否因此改变看法？

可落地 Skill：

- Planner：维护 `consequence_ledger`。
- Writer：写后果如何干扰下一步。
- Reviewer：低分必须退回 Planner/Writer。

## 9. 语言光泽和余味

定义：

- 语言是否有节制、具体、可记忆的质感，并在章尾留下人的余波，而不是只留下机制悬念。

好文本表现：

- 句子不炫技，但有贴身感。
- 章尾钩子之后仍能记住人物的某个状态。
- 总结句少，留白足。

坏文本表现：

- 大量“不是 X，是 Y”的主题阐释。
- 所有转折都被解释干净。
- 章尾只剩下一条机制悬念。

可审查问题：

- 章尾读者记住的是机制，还是一个人如何站在那里？
- 是否有可删的主题总结句？
- 是否存在过度整齐的逻辑句式？

可落地 Skill：

- Writer：控制总结句和解释句。
- Reviewer：标记“顺滑但无光泽”。
- Polisher：只在结构合格后处理节奏、留白和句式。

## 10. 是否仍像系统展示

定义：

- 文本是否仍让读者感觉在看机制 UI、设定说明、任务流程，而不是人在场景中承受变化。

好文本表现：

- 机制通过人的选择显形。
- 规则变化造成具体困境。
- 信息、钩子、爽点都被人物关系和生活状态承载。

坏文本表现：

- 积分、规则、纯度、阵法、回收标准像系统面板。
- 人物站在旁边解释机制。
- 每场戏的核心是“读者理解一个设定”，不是“人物不得不做一个选择”。

可审查问题：

- 如果删掉数值/术语，这场戏还剩人的冲突吗？
- 主角是在经历问题，还是在观察系统？
- 读者记住的是规则，还是人物付出的东西？

可落地 Skill：

- Planner：每个机制节点必须绑定人物选择和后果。
- Writer：让机制从行动中显形。
- Reviewer：作为总 gate，低于 3 分不得进入 Polisher。

## 推荐输出格式

Reviewer 可输出：

```yaml
human_texture_review:
  total_score: 0
  dimensions:
    character_credibility: 0
    private_want_shame_hesitation: 0
    emotional_realness: 0
    relationship_friction: 0
    scene_life_texture: 0
    non_instrumental_detail: 0
    natural_information_exposure: 0
    real_consequence: 0
    language_aftertaste: 0
    system_display_risk: 0
  hard_fail:
    - dimension: ""
      reason: ""
      return_to: "planner|writer|reviewer|polisher"
  required_revision_brief:
    - ""
```

## 与现有 skill-pack 的关系

- `webnovel_planner` 已能处理 cognitive advantage、rule fracture、revelation phases，但缺少人物私心、关系债、后果账。
- `webnovel_writer` 已能执行事件开章、规则破局、行动展示，但缺少场景生活阻力和非工具细节。
- `webnovel_reviewer` 已有 14 维审查，但主要看钩子、节奏、payoff、模板风险，Human Texture 需新增维度。
- `webnovel_polisher` 只能做轻增强，不能承担补人味结构的任务。
- `detect_webnovel_ai_flavor` 可作为语言风险检测，但不能替代本 rubric。
