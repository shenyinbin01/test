# Human Texture Packet Schema

## 设计目标

Human Texture packet 是写作约束，不是正文模板。它的作用是让一章从“机制展示故事”变成“人在机制压力下做选择，并留下关系、情绪、场景和后果”。

两轮 MVP 的原始 packet 字段较多，效果最好但也最容易模板化的部分，应压缩为 6 个核心字段：

1. `private_want`
2. `shame_or_avoidance`
3. `relationship_debt_change`
4. `scene_resistance`
5. `information_carrier`
6. `consequence_next_friction`

## 全局规则

- 不要求每章 6 个字段全部显性出现。
- 每章最多选择 2-3 个主字段重点执行。
- 字段是 Planner / Writer / Reviewer 之间传递的约束，不是正文可见小标题。
- Writer 不应逐项显性写出字段，而应把字段转成行为、停顿、误读、选择和后果。
- 不允许“为人味而人味”的装饰性细节。细节必须改变选择、遮蔽信息、制造阻力、暴露关系或留下后果。
- 如果一章是高机制推进章，也至少保留一个人的反应锚点，避免钩子完全压过人物。

## 建议最小结构

```yaml
human_texture:
  focus_fields:
    - relationship_debt_change
    - information_carrier
  private_want: ""
  shame_or_avoidance: ""
  relationship_debt_change: ""
  scene_resistance: ""
  information_carrier: ""
  consequence_next_friction: ""
```

## 字段选择策略

| 场景类型 | 推荐主字段 | 说明 |
| --- | --- | --- |
| 关系破裂 / 误会 / 欺瞒 | `private_want`, `shame_or_avoidance`, `relationship_debt_change` | 让人物不是“解释剧情”，而是在关系压力下做不完美选择。 |
| 信息露出 / 规则公布 / 设定揭示 | `scene_resistance`, `information_carrier`, `consequence_next_friction` | 让信息从具体人和制度缝隙里出来，并付出代价。 |
| 情绪后果 / 代价残留 | `shame_or_avoidance`, `relationship_debt_change`, `consequence_next_friction` | 让情绪改变下一步行动，而不是被一句“他很痛苦”消费掉。 |
| 章尾钩子 | `private_want`, `relationship_debt_change`, `consequence_next_friction` | 保留大钩子，同时留下人物承受后的余味。 |
| 战斗 / 规则破局 | `private_want`, `scene_resistance`, `consequence_next_friction` | 不削弱爽点，但让破局带来人的损失或关系压力。 |

## 字段定义

### 1. `private_want`

| 项 | 设计 |
| --- | --- |
| 定义 | POV 人物在公开目标之外，不愿完全承认的私人欲望。它可以很小，例如想被某人相信、想保住面子、想少欠一个人情。 |
| 解决的问题 | 人物像功能件，只负责推动机制、解释规则、完成任务。 |
| 好例子 | 主角明知应该解释，却先想保住“自己从未失控”的形象，因此避开柳青砚的眼神。 |
| 坏例子 | “他想赢”“他想保护大家”“他想变强”。这些只是主线目标，不是私人欲望。 |
| 适用场景 | 关系节点、选择节点、欺瞒节点、情绪代价节点、章尾余味节点。 |
| 不适用场景 | 纯转场、纯动作执行、没有人物选择空间的短功能段。 |
| Writer 是否需要显性写出 | 不需要。正文应通过动作、迟疑、借口、避让或优先级表现。 |
| Reviewer 如何检查 | 问：这个人物除了完成剧情功能，还在悄悄想保住什么或得到什么？这个欲望是否改变了他的一次选择？ |
| 模板化风险 | 容易变成“他其实只是想被理解”之类的心理口号。 |
| 压缩规则 | 一章只保留一个私人欲望；必须能落到一个可见选择上，否则删除。 |

### 2. `shame_or_avoidance`

| 项 | 设计 |
| --- | --- |
| 定义 | 人物不愿面对的羞耻、亏欠、害怕被看穿之处，以及因此产生的回避行为。 |
| 解决的问题 | 情绪空心化，人物只有痛苦标签，没有情绪后的行为后果。 |
| 好例子 | 主角被柳青砚发现撒谎后，没有立刻解释，而是先把话题推到任务上，因为承认会让他显得卑劣。 |
| 坏例子 | “他羞愧万分”“她非常失望”“空气变得沉重”。这些只命名情绪，没有回避动作。 |
| 适用场景 | 撒谎、失败、误判、背叛、亏欠、求助失败、被旁人看见软弱。 |
| 不适用场景 | 情绪不该成为重点的高速机制段；角色确实没有可回避之事的段落。 |
| Writer 是否需要显性写出 | 不需要。可以让人物换话题、沉默、错误补偿、过度解释、选择更糟的行动。 |
| Reviewer 如何检查 | 问：情绪是否导致人物做了一个不完美选择？如果删掉情绪句，行为还会不会变？ |
| 模板化风险 | 容易写成“低头”“握拳”“苦笑”等万能肢体表演。 |
| 压缩规则 | 避免连续心理独白。最多选择一个回避动作，并让它影响对话或下一步剧情。 |

### 3. `relationship_debt_change`

| 项 | 设计 |
| --- | --- |
| 定义 | 本章某段关系账的变化：欠下、赖掉、误伤、补偿失败、暂时信任、信任裂缝扩大。 |
| 解决的问题 | 人物关系缺少黏性，冲突当场发生当场抹平，下一章不背债。 |
| 好例子 | 柳青砚没有原谅主角，只把必要信息说完，并要求他下次不要再把她当成“可牺牲的知情者”。 |
| 坏例子 | “两人之间的信任产生裂痕”但后续对话、行动、资源分配都没有变化。 |
| 适用场景 | 队友关系、师徒关系、敌友暧昧、误会和解、合作交易、救援亏欠。 |
| 不适用场景 | 没有重要关系人的孤立探索段；仅用于一次性路人而不进入后续。 |
| Writer 是否需要显性写出 | 不需要写成总结句。应表现为称呼变化、信息保留、站位变化、资源分配、下次合作门槛。 |
| Reviewer 如何检查 | 问：这段关系在段落结束时和开始时有什么不同？下一章需要继承什么摩擦？ |
| 模板化风险 | 容易把所有关系都写成“信任-破裂-修复”的单线账本。 |
| 压缩规则 | 每章最多追踪 1-2 条关系债；关系变化必须有后续使用计划。 |

### 4. `scene_resistance`

| 项 | 设计 |
| --- | --- |
| 定义 | 场景本身对人物目标造成的具体阻力，包括生活秩序、制度规则、人群视线、噪声、排队、资源短缺、空间限制。 |
| 解决的问题 | 场景缺少生活质感，只像信息触发器或舞台背景。 |
| 好例子 | 饭堂里矿奴为抢发霉饼子挤成一团，主角想打听矿洞消息，却因为白牌监工巡桌，只能从别人漏掉的半句话里拼出信息。 |
| 坏例子 | “饭堂很嘈杂，很有生活气息”，随后信息仍由主角直接读懂规则或旁白公告。 |
| 适用场景 | 饭堂、矿洞、集会、执事堂、交易、审讯、赶路、战后清点。 |
| 不适用场景 | 完全抽象的内心推演段；场景阻力会打断必要高速节奏的极短钩子段。 |
| Writer 是否需要显性写出 | 需要写出阻力本身，但不要解释“这是生活质感”。阻力应妨碍行动或改变信息获取方式。 |
| Reviewer 如何检查 | 问：场景是否让人物更难完成目标？如果删掉场景细节，剧情是否仍完全不变？ |
| 模板化风险 | 容易堆杂物、气味、人群，形成装饰。 |
| 压缩规则 | 只保留一个与目标冲突相关的场景阻力；无功能细节必须删。 |

### 5. `information_carrier`

| 项 | 设计 |
| --- | --- |
| 定义 | 信息通过谁、什么物件、什么制度缝隙、什么误读或什么代价露出，而不是由旁白或权威公告直接说明。 |
| 解决的问题 | 信息公告化，设定像系统说明，人物像在读底层源码。 |
| 好例子 | 规则不是执事完整宣布，而是半张罚牌、被扣掉的饭食、老矿奴的沉默和监工的例外处理拼出。 |
| 坏例子 | “根据宗门规定，所有弟子必须……”后接完整规则说明；或者主角无代价直接看穿全部真相。 |
| 适用场景 | 规则公布、设定揭示、能力机制、敌方计划、制度漏洞、阵法线索。 |
| 不适用场景 | 读者必须瞬间理解的基础安全信息；已经多次铺垫、可快速收束的信息。 |
| Writer 是否需要显性写出 | 需要写出载体，但不要写成“信息载体是”。载体应自然存在于场景或人物行动中。 |
| Reviewer 如何检查 | 问：信息来自哪里？谁因此付出代价、误读、隐瞒或暴露立场？是否仍像公告？ |
| 模板化风险 | 容易固定成“路人闲聊透露一切”。 |
| 压缩规则 | 一个信息点只选一个主载体；不要为自然露出而绕远路解释。 |

### 6. `consequence_next_friction`

| 项 | 设计 |
| --- | --- |
| 定义 | 本章选择、破局、信息获取或情绪处理，给下一章留下的具体摩擦。 |
| 解决的问题 | 代价没有真实后果，爽点和钩子之后一切清零。 |
| 好例子 | 主角拿到矿洞线索，但因此欠下老矿奴一次沉默；下一章他必须在救人和保密之间选。 |
| 坏例子 | “他付出了沉重代价”但资源、人际、身体、制度追踪都没有变化。 |
| 适用场景 | 章尾、破局、交易、欺瞒、救援、信息交换、情绪爆发。 |
| 不适用场景 | 独立短插曲；下一章不继承此线的段落。 |
| Writer 是否需要显性写出 | 可以轻写出一个未完成动作、未还债、被看见的痕迹或下一章阻力，不需要总结成主题句。 |
| Reviewer 如何检查 | 问：下一章多了什么麻烦？谁记住了什么？什么资源少了、关系冷了、制度盯上了？ |
| 模板化风险 | 容易每章都变成“有人误会他”或“关系更糟”。 |
| 压缩规则 | 只留下一个最能推动下一章的摩擦；不要堆多条后果账。 |

## Reviewer 的整体检查句

Reviewer 不需要把 6 个字段全部打满，只需检查 `focus_fields` 是否被正文有效执行：

- 字段是否改变了人物选择？
- 字段是否改变了信息露出方式？
- 字段是否改变了关系或下一章摩擦？
- 字段是否保留了原有网文推进功能？
- 字段是否显性模板化，像在逐项完成表格？

如果答案只停留在心理口号、抽象总结或装饰细节，应退回 Planner 或 Writer，而不是交给 Polisher。
