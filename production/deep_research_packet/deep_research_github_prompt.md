# Deep Research GitHub URL Prompt

下面这段 prompt 适合项目负责人直接复制给 Deep Research。使用场景是只提供一个 GitHub URL，例如：

```text
https://github.com/shenyinbin01/test/blob/planning/deep-research-github-entry-v0/production/deep_research_packet/DEEP_RESEARCH_ENTRYPOINT.md
```

## 可复制文本

```text
请基于我提供的 GitHub 入口页，做一份关于“工程化网文创作系统”的深度研究报告。

请先打开这个入口页：

https://github.com/shenyinbin01/test/blob/planning/deep-research-github-entry-v0/production/deep_research_packet/DEEP_RESEARCH_ENTRYPOINT.md

请按入口页中的推荐读取顺序读取资料包。不要读取整个仓库，除非资料包引用某个文件且确有必要。请不要要求我上传小说正文、raw corpus 或原作原文。

资料包内容是当前项目判断，不是外部研究结论。请基于资料包做外部研究、理论对标、工程对标和批判性审计。不要只顺着项目设想说好话。

请明确区分：

1. 项目资料包里的当前判断。
2. 你从外部资料得到的研究结论。
3. 你自己的推理建议。

研究重点包括：

- 当前分层是否合理，是否存在缺层、重叠或顺序错误。
- 人物质感（Human Texture）、作品声音 / 作者站位（Work Voice / Narrative Stance）、角色主动感 / 行动主权（Character Agency）、活人浅痕 / 活人漏痕（Live Leakage）、读者代入感 / 读者解压（Reader Immersion）、角色驱动叙事引擎（Agentic Narrative Engine）分别对应哪些外部理论和相邻工程实践。
- 这套系统最大的工程风险和创作风险是什么。
- 如何避免工程化创作系统滑向更高级的套路生成器或内容农场。
- 哪些 MVP 应该优先做。
- 如果只能做一个 1-scene MVP，应该怎么设计。

禁止事项：

- 不要生成小说正文。
- 不要模仿具体作者。
- 不要要求上传原文语料。
- 不要把目标理解成作者指纹或作者复刻。
- 不要只给泛泛写作建议。
- 不要把网文简单等同于低质套路内容。
- 不要否认爽点、节奏、钩子在类型文中的必要性。
- 不要把工程化与真诚创作简单对立。
- 不要把角色驱动叙事引擎（Agentic Narrative Engine）理解成自由多 agent 聊天跑飞。
- 不要把活人浅痕 / 活人漏痕（Live Leakage）理解成多加细节。
- 不要把读者代入感 / 读者解压（Reader Immersion）理解成解释读者应该怎么想。
- 不要把作品声音 / 作者站位（Work Voice / Narrative Stance）理解成作者出来讲课。
- 不要把角色主动感 / 行动主权（Character Agency）理解成主角永远主动出击。

期望输出：

1. 执行摘要。
2. 项目框架评估。
3. 每个模块的理论参照。
4. 每个模块的相邻工程实践。
5. 主要风险清单。
6. MVP 优先级建议。
7. 推荐验证实验。
8. 反对意见与失败可能。
9. 可纳入路线图的建议。
10. 参考来源清单，并在正文中引用来源。
```
